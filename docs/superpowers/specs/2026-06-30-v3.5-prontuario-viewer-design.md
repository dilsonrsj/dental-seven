## 1. Objetivo

Completar o módulo **Prontuário** além do upload (v2.5): **visualizar documentos no app**, registrar **evolução clínica** e **gerar documentos** (receita, atestado, guia) com assinatura visual do dentista — sem ICP-Brasil e sem WhatsApp real (v6).

---

## 2. Decisões

| # | Decisão |
|---|---------|
| 1 | Entrega em **3 ondas** no mesmo plano (viewer → evolução → documentos clínicos) |
| 2 | **Viewer PDF:** `iframe` com signed URL (reutiliza `getDocumentDownloadUrl`) — sem `react-pdf` na v3.5 |
| 3 | **Viewer imagem:** modal lightbox com zoom simples (CSS/`transform`) |
| 4 | Rota viewer: `/pacientes/[id]/prontuario/[documentId]` ou modal na lista — **modal na lista** (menos rotas) |
| 5 | **Evolução clínica:** tabela `patient_clinical_notes` (`clinic_id`, `patient_id`, `appointment_id` nullable, `author_id`, `body`, `created_at`) |
| 6 | **Dentistas:** migration `009` — `cro`, `specialty`, `signature_storage_path` (bucket `clinic-assets` ou subpasta em `patient-documents` → usar bucket **`clinic-assets`** privado) |
| 7 | **Templates clínicos:** 3 templates fixos em código (receita, atestado, guia) — sem editor visual |
| 8 | **PDF gerado:** HTML + `window.print()` + salvar cópia em `patient_documents` com `source: 'generated'` via geração server-side simples (texto/HTML exportado como PDF usando biblioteca leve **`pdf-lib`** apenas na geração server) **OU** salvar HTML snapshot + impressão client — **decisão: `pdf-lib` no server action** para arquivo persistido |
| 9 | **Envio WhatsApp v3.5:** mensagem outbound simulada na thread do paciente: *"Documento [título] disponível na clínica"* — sem anexo binário até v6 |
| 10 | Branch: **`feat/v2` apenas** — `main`/Vercel demo intocável |
| 11 | Módulo `prontuario` continua gate por `clinic_modules` |

## 3. Escopo por onda

### Onda 1 — Viewer inline (MVP v3.5)

| Item | Detalhe |
|------|---------|
| Botão **Visualizar** na lista de documentos | Abre modal fullscreen |
| PDF | `iframe` com signed URL (60–120s) |
| JPG/PNG | `<img>` + zoom +/- no modal |
| Ações no modal | Fechar, Baixar (link existente) |
| Mobile | Modal responsivo, scroll interno |

### Onda 2 — Evolução clínica + perfil dentista

| Item | Detalhe |
|------|---------|
| Migration `009_dentists_clinical_notes.sql` | Colunas em `dentists` + tabela `patient_clinical_notes` + RLS |
| UI na aba Prontuário | Seção "Evolução" — lista cronológica + formulário novo registro |
| Vínculo opcional | Select de consulta recente do paciente |
| Config dentista | Em `/configuracoes` ou modal: CRO, especialidade, upload assinatura (PNG) |
| Permissões | `clinic_admin` e `dentist` criam; paywall bloqueia escrita |

### Onda 3 — Documentos clínicos gerados

| Item | Detalhe |
|------|---------|
| Botão **Novo documento** | Escolha: Receita · Atestado · Guia de exame |
| Formulário por template | Campos mínimos (medicamentos/texto, dias afastamento, exames solicitados) |
| Geração | Server action monta PDF com `pdf-lib` + dados clínica/dentista/paciente + imagem assinatura |
| Persistência | Upload PDF gerado → `patient_documents` (`source: 'generated'`) |
| Impressão | `window.print()` em preview HTML antes de salvar **ou** abrir PDF gerado |
| WhatsApp simulado | Action `sendDocumentToWhatsAppThread(patientId, documentId)` — mensagem texto na thread demo |

## 4. Modelo de dados (novo)

```sql
-- dentists (alter)
alter table dentists add column cro text;
alter table dentists add column specialty text;
alter table dentists add column signature_storage_path text;

create table patient_clinical_notes (
  id uuid primary key default gen_random_uuid(),
  clinic_id uuid not null references clinics(id) on delete cascade,
  patient_id uuid not null references patients(id) on delete cascade,
  appointment_id uuid references appointments(id) on delete set null,
  author_id uuid references profiles(id) on delete set null,
  body text not null,
  created_at timestamptz not null default now()
);
```

Bucket **`clinic-assets`**: path `{clinic_id}/signatures/{dentist_id}.png`

## 6. Abordagens consideradas

| Abordagem | Prós | Contras | Veredito |
|-----------|------|---------|----------|
| **A — Viewer iframe + pdf-lib server** | Poucas deps, reusa Storage | PDF complexo em pdf-lib | **Recomendada** |
| B — react-pdf viewer | UX rica | Bundle grande, mais estado | Adiar |
| C — Só download (status quo v2.5) | Zero trabalho | Não cumpre v3.5 | Rejeitada |
